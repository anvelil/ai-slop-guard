## Descriptions

### ASG001 — Unused imports (stable)

An import is present but never referenced in the file.

**Why:** Unambiguous and always safe to act on: Python's ast resolves it exactly, with no guessing about scope or usage elsewhere.

**Example (flagged):**
```
import os
# os is never used below
```

**Counterexample (not flagged):**
```
import os
os.environ['DEBUG'] = '1'  # used -> not flagged
```

**Tradeoffs:** No known false positives on plain import/from-import. Python's __all__ re-exports (string literals only -- `__all__ = ["x"]` or `+= [...]`) are recognized as a use since 0.3.0. Dynamically built __all__ (e.g. a list comprehension) and TYPE_CHECKING-only imports used solely in string annotations are still not understood -- see docs/known-limitations.md.

### ASG002 — Dead code (module-level candidate) (experimental)

A module-level function is defined but never called anywhere else in the same file. Class methods, decorated functions, and pytest test_* functions are excluded.

**Why:** Dead code is a real cost -- it's read, half-understood, and sometimes 'fixed' by someone who doesn't realize nothing calls it. Flagging module-level candidates is cheap and, once decorators/tests are excluded, has a low false-positive rate on real code.

**Example (flagged):**
```
def unused_helper():
    return 42
# never called anywhere in the file
```

**Counterexample (not flagged):**
```
@app.route('/')
def index():
    ...
# decorated -> not flagged, framework calls it indirectly
```

**Tradeoffs:** Status is 'experimental' rather than 'stable': a benchmark against Flask's own tutorial app found a 34/34 false-positive rate before the decorator/test_* exclusion was added (see benchmarks/README.md), and this class of scope-narrowing may not be exhaustive yet -- other frameworks may have their own indirect-call conventions not yet excluded. It can never see cross-file calls, so every finding needs manual verification regardless.

### ASG003 — Catch-all / empty exception handling (stable)

A bare or broad except/catch block whose body only passes, continues, or logs, silently swallowing the error.

**Why:** A broad except that does nothing observable hides bugs instead of fixing them -- the one shape of exception handling that's actively worse than no handling at all, since it also suppresses the traceback.

**Example (flagged):**
```
try:
    risky()
except Exception:
    pass
```

**Counterexample (not flagged):**
```
try:
    risky()
except Exception as e:
    raise RuntimeError('risky() failed') from e
# re-raises with context -> not flagged
```

**Tradeoffs:** Deliberately conservative: only empty/pass/continue/single-log bodies trigger it, so a broad except that does real cleanup (closing a resource, rolling back a transaction) alongside more statements is not flagged, even though it may still be worth a human look.

### ASG004 — Duplicated logic (planned)

Near-identical logic exists elsewhere in the project under a different name. Requires whole-project context the script does not have.

**Why:** Two near-identical functions with different names is the clearest sign an agent didn't look at the existing codebase before writing new code, and it doubles the maintenance surface for one piece of logic.

**Example (flagged):**
```
function formatUserDate(d) { return d.toISOString().split('T')[0]; }
function formatOrderDate(d) { return d.toISOString().split('T')[0]; }
```

**Counterexample (not flagged):**
```
function formatDate(d) { return d.toISOString().split('T')[0]; }
// single shared implementation -> nothing to flag
```

**Tradeoffs:** Manual for now — no design work has started on making this script-verifiable. Whole-project indexing plus some notion of semantic similarity would be needed; not attempted yet.

### ASG005 — Unnecessary defensive checks (planned)

A guard against null/undefined for a value whose type already rules that out. Requires type information the script does not have.

**Why:** A check that can never trigger protects against nothing, adds noise to every diff that touches the function, and creates false confidence that the value was actually validated.

**Example (flagged):**
```
function getFullName(user: User): string {
  if (!user) return '';
  return `${user.firstName} ${user.lastName}`;
}
```

**Counterexample (not flagged):**
```
function getFullName(user: User | null): string {
  if (!user) return '';
  return `${user.firstName} ${user.lastName}`;
}
// user can genuinely be null here -> not unnecessary
```

**Tradeoffs:** Manual for now. Would need a resolved type checker's output, not syntax alone — nothing built yet.

### ASG006 — Comments that restate the code (planned)

A comment describes what the code does instead of why, adding no information beyond the code itself. Judgment call.

**Why:** A comment that just restates the next line adds reading time without adding information, and worse, rots silently -- nobody updates a comment that never had to be true.

**Example (flagged):**
```
// increment counter by one
counter += 1;
```

**Counterexample (not flagged):**
```
// compensate for the off-by-one in the API, which counts from zero
counter += 1;
// explains 'why' -> not flagged
```

**Tradeoffs:** Manual, and likely to stay that way — this is closer to a judgment call than a rule.

### ASG007 — Leftover debug output (stable)

A print()/console.log()-style call left in code that is not deliberate logging through the project's logger.

**Why:** Debug prints are the most common single artifact of iterative, exploratory agent work -- cheap to add while debugging, easy to forget to remove, and each one is a small, avoidable embarrassment in a diff.

**Example (flagged):**
```
def add(a, b):
    result = a + b
    print(result)
    return result
```

**Counterexample (not flagged):**
```
def add(a, b):
    result = a + b
    print(result)  # slop-guard: ignore -- CLI tool's intended output
    return result
```

**Tradeoffs:** AST-based for Python, so it won't match the word 'print' inside a string or docstring -- but it also can't distinguish 'genuinely forgotten' from 'temporarily commented out for a debugging session', both get flagged the same way.

### ASG008 — Hallucinated API usage (planned)

Code calls a method/attribute/parameter that does not exist on the referenced type or library version. Not implemented: reliably detecting this needs resolved type information or the actual library's API surface — single-file static analysis alone isn't enough.

**Why:** One of the most common and costly agent mistakes -- confidently calling a plausible-sounding method that was never real, or that existed in a different library version -- and one of the hardest to catch without running the code.

**Example (flagged):**
```
response = requests.get(url)
data = response.to_json()  # requests.Response has .json(), not .to_json()
```

**Counterexample (not flagged):**
```
data = response.json()  # correct API -> nothing to flag, once this rule exists
```

**Tradeoffs:** Registered so it isn't forgotten. No design work has started; a naive implementation would guess at plausible method names, which isn't good enough to ship.
