# ИИ-слоп в программировании: почему мусорный код от нейросетей опасен для ваших проектов


## Что такое ИИ-слоп

Обычно под термином «ИИ-слоп» (AI slop) понимают бессмысленные сгенерированные тексты, картинки или дешевые видеоролики, которыми нейросети заполняют интернет. Это контент, который выглядит правдоподобно на первый взгляд, но не несет никакой ценности.

В разработке программного обеспечения появился свой аналог — кодовый слоп.

Кодовый ИИ-слоп — это синтаксически корректный, проходящий тесты код, который решает поставленную задачу, но при этом перегружен незаметным мусором. Это не баги в привычном понимании. Линтер форматирования не выдаст предупреждений, тесты будут зелеными, а программа запустится. Но внутри останется шлейф из неиспользуемых импортов, забытых отладочных выводов, мертвых функций и пустых конструкций обработки ошибок.


## Почему ИИ-слоп опаснее обычных багов

Обычный баг проявляет себя быстро: приложение падает, тест выдает ошибку, пользователь видит сломанный интерфейс. Баг заставляет разработчика вмешаться и исправить код. 

Слоп ведет себя иначе. Он не ломает систему сразу, а накапливается постепенно, отравляя проект изнутри.

### 1. Отличие от классического техдолга
Классический технический долг — это осознанный компромисс. Разработчик решает: «Сейчас мы напишем простую синхронную интеграцию, чтобы успеть к релизу, а через месяц перепишем на очереди». У техдолга есть архитектурное или коммерческое обоснование.

ИИ-слоп — это чистый энтропийный шум. У него нет никакой цели. Нейросеть оставила неиспользуемую библиотеку не потому, что спешила к релизу, а просто потому, что забыла её убрать в процессе рефакторинга. Это случайный мусор, который не приносит проекту никакой пользы, но требует ресурсов на поддержку.

### 2. Эффект вареной лягушки в кодовой базе
Когда разработчик пишет код сам, он понимает назначение каждой строки. Внедрение ИИ-помощников (агентов, автокомплитов) меняет этот паттерн. Разработчик начинает принимать диффы от нейросети крупными блоками. 

Если код работает, человек склонен доверять ИИ. Мелкий мусор игнорируется. Со временем кодовая база забивается сотнями неиспользуемых строк. Новый разработчик, приходя в проект, тратит время на чтение мертвых функций, пытаясь понять, в каких сценариях они вызываются, хотя они не вызываются вообще. Теряется доверие к коду: разработчики перестают понимать, какие части системы действительно важны.


## Конкретные проявления кодового слопа

Вот четыре основных паттерна, которые ИИ-помощники оставляют в коде чаще всего.

### Паттерн 1: Забытый отладочный вывод
В процессе написания кода ИИ постоянно добавляет отладочные конструкции для проверки промежуточных данных. После исправления проблемы эти строки часто остаются в финальном коде.

*Пример на Python:*
```python
def process_order(order_id):
    order = db.get_order(order_id)
    print("DEBUG ORDER:", order)  # Забытый принт, который улетит в логи сервера
    return calculate_price(order)
```

В масштабах продакшна такие принты не просто засоряют логи, но и могут приводить к утечке конфиденциальной информации пользователей.

### Паттерн 2: Неиспользуемые импорты
ИИ генерирует код итерациями. На одном шаге он решает попробовать библиотеку `math` или `re`, добавляет импорт, а на следующем шаге переписывает алгоритм без их использования. Импорт в начале файла остается.

*Пример на TypeScript:*
```typescript
import { useState, useEffect, useMemo, useCallback } from 'react'; // useMemo и useCallback больше не используются

export const UserProfile = ({ userId }) => {
  const [user, setUser] = useState(null);
  useEffect(() => {
    fetchUser(userId).then(setUser);
  }, [userId]);
  return <div>{user?.name}</div>;
};
```

### Паттерн 3: Мертвые функции-кандидаты
В процессе перебора решений ИИ может написать вспомогательную функцию, затем поменять логику и оставить старую функцию нетронутой. Она стоит в файле, увеличивает объем кода и путает тех, кто будет проводить ревью или дорабатывать логику.

*Пример на Python:*
```python
def _legacy_format_date(dt):
    # Эта функция больше не вызывается в модуле, но ИИ оставил её на всякий случай
    return dt.strftime("%Y-%m-%d")

def get_formatted_now():
    return datetime.now().isoformat()
```

### Паттерн 4: Молчаливое подавление ошибок
Самый опасный вид слопа. Чтобы избавиться от ошибок времени выполнения во время тестирования, ИИ оборачивает подозрительный код в блоки перехвата исключений, которые ничего не делают.

*Пример на JavaScript:*
```javascript
try {
  initializePaymentGateway();
} catch (error) {
  // Ошибка проглочена без логирования. Система продолжит работу, 
  // но платежи будут падать без объяснения причин.
}
```


## Почему ИИ генерирует слоп и почему его пропускают на ревью

Причина генерации слопа кроется в итеративном характере работы нейросетей. Контекстное окно модели сосредоточено на решении текущей задачи (например, «пофиксить ошибку деления на ноль»). Исправив ошибку, модель считает задачу выполненной. У нее нет естественного рефлекса вернуться назад, сделать `git diff` и удалить все временные конструкции.

На этапе ревью человек ищет крупные архитектурные проблемы, уязвимости или критические баги. Лишний импорт или мертвая функция на фоне сотен строк изменений выглядят как белый шум. Ревьюер просто не обращает на них внимания.


## Решение: автоматизация борьбы со слопом

Просить ИИ «проверить код перед сдачей» помогает плохо: модель проверяет его своим же контекстным окном с теми же слепыми пятнами. Для решения этой проблемы нужен независимый внешний контроль.

Для этого я разработал `slop-guard` — статический анализатор без внешних зависимостей, который работает локально и выявляет именно признаки ИИ-слопа.

Инструмент проверяет кодовую базу на:
- Неиспользуемые импорты (с корректной обработкой Python `__all__`).
- Неиспользуемые функции уровня модуля.
- Пустые и немые перехваты исключений.
- Оставленные отладочные принты.

Пример вывода:
```bash
slop-guard src/
```

```
ASG001  Unused import "json"  (src/main.py:4)
ASG007  Leftover debug output  (src/main.py:22)

ai-slop-guard: 2 finding(s)
```

Репозиторий проекта: [github.com/anvelil/ai-slop-guard](https://github.com/anvelil/ai-slop-guard)

Установка:
```bash
pip install git+https://github.com/anvelil/ai-slop-guard.git
```

Инструмент легко настраивается как шаг в CI/CD или локальный pre-commit хук, гарантируя, что мусорный код от нейросетей не попадет в ваш репозиторий.
