# Скрытая угроза ИИ-слопа: почему код от нейросетей опасен для ваших проектов и как это решить


## Что такое ИИ-слоп и в чём его опасность

Когда мы говорим о проблемах кода от ИИ, обычно вспоминают явные галлюцинации: вызов несуществующих функций или грубые логические ошибки. Но такие проблемы легко заметить — тесты падают, программа не собирается, компилятор ругается. 

Гораздо опаснее другое явление — ИИ-слоп (AI slop). Это код, который успешно работает, проходит тесты и выполняет задачу, но при этом содержит скрытый мусор:
- Забытые вызовы `print()` и `console.log()` после отладки.
- Неиспользуемые импорты библиотек, которые остались после промежуточных вариантов решения.
- Мертвые функции, которые агент написал в процессе работы, но потом перестал вызывать.
- Пустые блоки `except: pass` или `catch {}`, которые просто глушат ошибки.

На первый взгляд это кажется мелочью. Лишний импорт не сломает продакшн, а мертвая функция просто займет лишние 10 строк. Но опасность ИИ-слопа проявляется со временем.

**Накопление технического долга.** ИИ генерирует код в десятки раз быстрее человека. Если каждый раз оставлять в файлах по 5% мусора, проект быстро превратится в болото. Через месяц разобраться в структуре станет невозможно: вы будете читать код функций, которые вообще не вызываются, и пытаться понять их смысл.

**Сложный поиск реальных багов.** Пустой `except Exception: pass` — это худшее, что можно сделать для отладки. Если ИИ столкнется с ошибкой подключения к базе данных или сети, он просто проигнорирует её и вернет пустой результат. Вы потратите часы на поиск проблемы, потому что код работает тихо и не выдает никаких логов или исключений.

**Засорение логов.** Забытые отладочные принты в цикле могут перегрузить систему сбора логов и затруднить анализ реальных инцидентов.


## Почему ИИ оставляет мусор

Причина в природе работы современных ИИ-агентов. Они работают итеративно. Агент получает задачу, пишет код, видит ошибку, исправляет её, и так по кругу.

На каждой итерации контекст ИИ фокусируется только на текущей микро-задаче. Он не оглядывается назад и не анализирует весь файл целиком на предмет чистоты. У него нет внутреннего триггера провести финальную уборку: удалить неиспользуемый импорт или стереть отладочный вывод, который помог ему шаг назад. 


## Почему это пропускают на ревью

Ревьюер ищет бизнес-логику, уязвимости безопасности и архитектурные несоответствия. Когда вам на ревью приходит диф на 300 строк от ИИ, вы физически не можете проверить, используется ли каждый импортированный модуль или вызывается ли вон та маленькая вспомогательная функция. Внимание сосредоточено на главном алгоритме.


## Решение: автоматический контроль ИИ-слопа

Чтобы решить эту проблему, нужен инструмент, который автоматически проверяет код именно на наличие ИИ-слопа перед коммитом. Обычные линтеры решают другие задачи (форматирование, синтаксис), но они часто пропускают неиспользуемые локальные функции или отладочный вывод.

В качестве решения я разработал `slop-guard` — легкий статический анализатор, написанный на чистом Python без внешних зависимостей. Его задача — быстро проверять файлы и находить именно те паттерны, которые чаще всего оставляет ИИ.

Инструмент проверяет код по встроенному реестру правил:
- **ASG001 (Unused imports)**: находит импорты, которые не используются (с поддержкой корректной обработки `__all__` в Python).
- **ASG002 (Dead code)**: детектирует неиспользуемые функции уровня модуля.
- **ASG003 (Empty exceptions)**: выявляет пустые блоки catch и except, которые скрывают ошибки.
- **ASG007 (Leftover debug)**: находит забытые принты и консольные логи.

Пример работы в терминале:

```bash
slop-guard src/
```

Вывод:
```
ASG001  Unused import "os"  (src/api.py:3)
  Reason: 'os' is imported on line 3 but never referenced.
  Suggested fix: Remove the import of 'os'.

ASG007  Leftover debug output  (src/api.py:47)
  Reason: print(...) call left in code.
  Suggested fix: Remove it, or route through the project's logger.

ai-slop-guard: 2 finding(s).
```

## Как внедрить это в рабочий процесс

Линтер отлично работает в связке с методологией разработки. Оптимальный пайплайн работы выглядит так:

1. **Анализ и планирование**: ИИ анализирует код и строит план без написания логики.
2. **Генерация и мысленная компиляция**: создание кода и базовая проверка связей.
3. **Техническая проверка**: проверка синтаксиса и автоматический запуск `slop-guard`.
4. **Ревью и рефакторинг**: ручной аудит сложных правил (например, проверка дублирования логики) и исправление найденных замечаний.
5. **Финальный аудит**: повторный запуск `slop-guard` для контроля, что в процессе исправления ошибок не появился новый мусор.

Инструмент можно установить одной командой:

```bash
pip install git+https://github.com/anvelil/ai-slop-guard.git
```

Его легко добавить в ваш список pre-commit хуков или настроить как шаг в GitHub Actions CI. Это гарантирует, что ни один коммит с ИИ-слопом не попадет в основную ветку вашего репозитория.
